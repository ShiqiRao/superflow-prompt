## ğŸ”€ Pull Request Process

1. **Fork the repository** and create your branch from `main`:
   ```bash
   git checkout -b feature/amazing-feature
   ```

2. **Make your changes**:
   - Follow the existing file structure
   - Maintain consistent formatting
   - If adding examples, use the template structure

3. **Test your changes**:
   - Ensure all markdown files render correctly
   - Check all links work
   - Proofread for typos

4. **Commit your changes**:
   ```bash
   git commit -m "Add: brief description of your changes"
   ```

   Use conventional commits:
   - `Add:` for new features
   - `Fix:` for bug fixes
   - `Update:` for improvements
   - `Docs:` for documentation changes

5. **Push to your fork**:
   ```bash
   git push origin feature/amazing-feature
   ```

6. **Open a Pull Request**:
   - Use a clear, descriptive title
   - Describe what you changed and why
   - Reference any related issues

## ğŸ“‹ Style Guide

### Markdown Formatting
- Use ATX-style headers (`#`, `##`, etc.)
- Add blank lines before and after headers
- Use `-` for unordered lists
- Keep lines under 120 characters when possible

### Content Guidelines
- **Be concise**: Short, actionable content is better
- **Be practical**: Include real examples and use cases
- **Be inclusive**: Use gender-neutral language
- **Be inspiring**: Maintain the motivational tone

### Example Structure
When creating examples, follow this structure:
1. Identity statement
2. Tasks table with status
3. Daily cycle section
4. Flow triggers
5. Journal entry
